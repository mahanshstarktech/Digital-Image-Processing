Why Image Compression?

Conceptual

The Size Problem

A standard uncompressed digital image consumes enormous storage. Consider:

  • A 1024×1024 grayscale image at 8 bits/pixel = 1 MB
  • A 1024×1024 RGB color image = 3 MB
  • A 4K video at 30fps = ~750 MB/second uncompressed

Image compression reduces this data by exploiting redundancy — information that is repeated, predictable, or imperceptible to the human eye.

Types of Redundancy

🔥 90% · Part A/B/C

Overview

Compression works by removing redundancy. There are three types of redundancy in digital images:

  1. Coding Redundancy — Inefficient use of bits in encoding
  2. Interpixel (Spatial/Temporal) Redundancy — Correlation between adjacent pixels
  3. Psychovisual Redundancy — Information the human eye cannot perceive

⭐⭐ 1. Coding Redundancy

Coding redundancy exists when a code (like 8-bit fixed-length) uses more bits than necessary. In an image, some gray levels occur more frequently than others. If frequent values get shorter codes (and rare values get longer codes), fewer total bits are needed.

Average Code Length
L_avg = Σ p(rₖ) · l(rₖ)
p(rₖ) = probability of gray level rₖ, l(rₖ) = code length assigned to rₖ
Entropy (Minimum Possible Average Code Length)
H = -Σ p(rₖ) · log₂[p(rₖ)] bits/pixel
Entropy H is the theoretical minimum. Any code with L_avg = H achieves zero coding redundancy.
Coding Redundancy
R_coding = L_natural - L_avg
L_natural = fixed natural code length (typically 8 bits for 256 gray levels)
📌 Example — Coding Redundancy

Suppose an image has 4 gray levels with probabilities:

Gray level  |  Probability  |  Natural code  |  Huffman code
  r₀ = 0   |    0.5        |    00          |    0    (1 bit)
  r₁ = 1   |    0.25       |    01          |    10   (2 bits)
  r₂ = 2   |    0.125      |    10          |    110  (3 bits)
  r₃ = 3   |    0.125      |    11          |    111  (3 bits)

Natural code: L_natural = 2 bits/pixel always

Huffman code: L_avg = 0.5×1 + 0.25×2 + 0.125×3 + 0.125×3 = 0.5 + 0.5 + 0.375 + 0.375 = 1.75 bits/pixel

Entropy: H = -(0.5·log₂0.5 + 0.25·log₂0.25 + 0.125·log₂0.125 + 0.125·log₂0.125)
= -(0.5×(-1) + 0.25×(-2) + 0.125×(-3) + 0.125×(-3)) = -(-0.5 - 0.5 - 0.375 - 0.375) = 1.75 bits/pixel

Coding Redundancy removed: 2 - 1.75 = 0.25 bits/pixel saved

⭐⭐ 2. Interpixel (Spatial) Redundancy

Adjacent pixels in a typical image tend to have similar or related values. In a smooth sky region, pixel values change very little from neighbor to neighbor. This correlation = interpixel redundancy.

Types:

  • Spatial redundancy: Correlation between neighboring pixels in the same frame
  • Temporal redundancy: Correlation between corresponding pixels in consecutive video frames (background doesn't change between frames)
  • Spectral redundancy: Correlation between color channels (R, G, B values of the same pixel are often similar)
💡
Quick Definition for Exam

Interpixel redundancy: "The correlation (statistical dependence) between neighboring pixels in a digital image. Adjacent pixels in smooth regions take similar values, making their individual storage redundant." Methods to exploit it: Run-Length Encoding (RLE), predictive coding, transform coding (DCT/JPEG).

⭐⭐ 3. Psychovisual Redundancy

The Human Visual System (HVS) does not perceive all visual information with equal sensitivity. Some information is redundant from the viewer's perspective — it can be removed without significantly affecting perceived image quality.

Characteristics of HVS that cause psychovisual redundancy:

  • Brightness sensitivity: HVS is more sensitive to brightness changes than color changes
  • Contrast sensitivity: HVS is less sensitive to high-frequency spatial patterns in uniform regions
  • Weber's Law: JND (Just Noticeable Difference) is proportional to background luminance — brighter areas can tolerate more quantization noise
  • Masking effect: Fine details in textured areas are harder to perceive than in smooth areas

This is exploited in lossy compression — discard what the eye can't see. JPEG uses psychovisual redundancy via its quantization table.

⚠️
Key Difference for Exam

Coding redundancy = mathematical/statistical redundancy (fixable by better coding). Interpixel redundancy = structural redundancy (fixable by predictive/transform coding). Psychovisual redundancy = perceptual redundancy (only removable in lossy compression, information is permanently lost).

📝 RTU PYQs — Redundancy
Q. What is data redundancy? Explain coding redundancy, interpixel redundancy, and psychovisual redundancy with examples.
June 2022 · Part B Q4 · 10 marks | Also: 2024 Part C Q2, 2025 Part A Q10
Data Redundancy in Image Compression:
If b represents bits in original image and b' in compressed image, then:
Compression ratio: Cr = b/b'
Relative redundancy: Rd = 1 - 1/Cr = 1 - b'/b

When Cr = 1 → Rd = 0 (no redundancy). When Cr >> 1 → Rd → 1 (high redundancy).

1. Coding Redundancy:
Standard 8-bit coding assigns equal 8 bits to every gray level regardless of frequency. A natural image typically has a non-uniform histogram — some gray levels appear much more than others. Using variable-length codes (shorter for frequent values, longer for rare ones) reduces the average code length.
Example: If gray level 0 (black) occurs 50% of the time and gets a 1-bit code instead of 8-bit, we save 7 bits for half the pixels.
Measured by: L_avg = Σ p(rₖ)·l(rₖ) vs entropy H = -Σ p(rₖ)·log₂[p(rₖ)]

2. Interpixel Redundancy:
Adjacent pixels are statistically correlated (neighboring pixels in sky, grass, wall tend to have similar values). Storing each pixel independently ignores this correlation.
Example: Row of 100 pixels all with value 150: storing 100 individual values is redundant; store "150 repeated 100 times" instead (RLE).
Removal methods: Run-length encoding, differential coding, transform coding (DCT).

3. Psychovisual Redundancy:
The HVS has limited ability to detect fine spatial details, especially in textured regions or bright areas. Information imperceptible to the eye is psychovisually redundant.
Example: A 256-level grayscale region where neighboring levels 127 and 128 cannot be distinguished can be quantized to fewer levels without visible degradation.
Removal method: Quantization (used in lossy JPEG). Once removed, this information cannot be recovered.

Compression Metrics & Formulas

85% · Essential Formulas

All Key Formulas (Memorise These!)

1. Compression Ratio
Cr = b / b' = (original bits) / (compressed bits)
Also written as n₁/n₂ or input_bits/output_bits. Higher = better compression.
2. Relative Data Redundancy
Rd = 1 - 1/Cr = 1 - b'/b
Range: [0, 1]. Rd=0 → no redundancy removed. Rd→1 → maximum compression. Rd=0.9 means 90% of data was redundant.
3. Average Code Length
L_avg = Σ p(rₖ) · l(rₖ) bits/pixel
Sum over all K gray levels. For natural code: L_natural = log₂(K) for K gray levels.
4. Entropy (Shannon)
H = -Σ p(rₖ) · log₂[p(rₖ)] bits/pixel
Theoretical minimum average code length. L_avg ≥ H always.
5. Coding Efficiency
η = H / L_avg × 100%
Huffman code achieves η closest to 100%. Natural code: η = H/L_natural (often 50–80%).
6. Coding Redundancy
R_coding = L_natural - L_avg (bits saved per pixel)
7. MSE and PSNR (Image Quality after Lossy Compression)
MSE = (1/MN) · Σ Σ [f(x,y) - f̂(x,y)]²
PSNR = 10 · log₁₀(255² / MSE) dB
f = original, f̂ = reconstructed. Higher PSNR = better quality. PSNR > 30dB is generally acceptable.
🧠
Quick Entropy Calculation Trick

For common probabilities, memorize these log₂ values:
log₂(0.5) = -1 | log₂(0.25) = -2 | log₂(0.125) = -3 | log₂(0.0625) = -4
For others: log₂(x) = ln(x)/ln(2) ≈ log₁₀(x)/0.301
log₂(0.4) = log₁₀(0.4)/0.301 = (-0.398)/0.301 ≈ -1.322
log₂(0.3) ≈ -1.737 | log₂(0.2) ≈ -2.322 | log₂(0.1) ≈ -3.322

Huffman Coding

🔥 95% — Most Asked Numerical

Core Idea

Huffman Coding is a lossless, variable-length coding technique that assigns the shortest codes to the most frequent symbols. It produces an optimal prefix-free code — no codeword is a prefix of another.

Key property: Huffman coding minimizes the average code length L_avg. It achieves L_avg = H (entropy) when all probabilities are negative powers of 2 (0.5, 0.25, 0.125, etc.).

Algorithm (must memorise for exam!)

1
List all symbols sorted by probability in descending order.
2
Combine the two symbols with lowest probability into a new combined node. Its probability = sum of the two.
3
Re-sort the list (insert combined node in correct position).
4
Repeat steps 2–3 until only one node remains (the root with prob=1.0).
5
Trace back from root to each symbol: assign 0 for the larger branch (or higher probability), 1 for the smaller branch at each merge.
⚡
Standard Convention Trick

RTU expects you to assign 0 to the higher probability branch and 1 to the lower at each split. This is the most common convention. Always state this in your answer. The resulting code is valid as long as it's prefix-free; exact bit assignments may vary but average code length must remain the same.

5. L_avg ≥ H (code cannot beat entropy)
6. Coding efficiency η ≤ 100%

Arithmetic Coding

🔥 95% — Appeared 2024 & 2025 Both!

Concept

Arithmetic coding represents an entire message as a single number in the range [0, 1). Unlike Huffman coding (which assigns a code to each symbol), arithmetic coding operates on the entire sequence, achieving compression closer to the entropy lower bound.

Key idea: Map the message to a unique sub-interval within [0, 1). The length of this interval decreases as more symbols are encoded. The final code is any number that falls within this interval (represented in binary).

Encoding Procedure

Each symbol is assigned a probability range (sub-interval of [0,1)). The intervals are cumulative and non-overlapping.

1
Set initial interval [low, high) = [0, 1). Range = 1.0.
2
For each symbol in the message, find its probability interval [cum_low, cum_high).
3
New high = old_low + range × cum_high. New low = old_low + range × cum_low. New range = new_high - new_low.
4
After all symbols, pick any number in [low, high) as the code.

LZW Coding (Lempel-Ziv-Welch)

80% · Always with Arithmetic PYQ

Core Idea

LZW coding is a lossless, dictionary-based compression algorithm. It does NOT need prior knowledge of symbol probabilities — it builds the dictionary dynamically as it processes the input.

Key property: Both encoder and decoder build the SAME dictionary independently — no dictionary transmission needed!

LZW Encoding Algorithm

1
Initialize dictionary with all single-character sequences. For a 256-character alphabet, entries 0–255 are pre-filled.
2
Read characters one at a time. Build string P = current longest match in dictionary.
3
When P + next_char (= PC) is NOT in dictionary: output code for P, add PC to dictionary, set P = next_char.
4
When P + next_char IS in dictionary: set P = PC and continue reading.
5
At end of input: output code for P.

LZW Decoding Algorithm

1
Initialize dictionary same way as encoder (single chars).
2
Read first code → output its string. Save as previous string (prevStr).
3
For each next code: if code is IN dictionary → output dict[code]. Add prevStr + first_char(dict[code]) to dictionary. prevStr = dict[code].
4
If code is NOT in dictionary (special case): entry = prevStr + prevStr[0]. Output it. Add to dictionary.

Run-Length Encoding (RLE)

70% · Part B

Concept

Run-Length Encoding (RLE) exploits interpixel redundancy. A "run" is a sequence of consecutive pixels with the same gray value. Instead of storing each pixel individually, RLE stores (value, count) pairs.

RLE Representation
Row = [(v₁, r₁), (v₂, r₂), (v₃, r₃), ...]
vᵢ = pixel value (gray level), rᵢ = run length (count of consecutive vᵢ pixels)
📌 RLE Example

Original row (8 pixels): 5, 5, 5, 8, 8, 8, 8, 5

RLE encoded: (5, 3), (8, 4), (5, 1) → 3 pairs = 6 values instead of 8

Compression for binary image:
Row: 0,0,0,0,1,1,1,0,0,1,1,1,1,0 → (4, 3, 2, 4, 1) = 5 counts [just counts since value alternates 0,1,0,1,0]

💡
When is RLE Effective?

RLE is best when runs are long (highly repetitive data). For binary (fax) images with long black/white runs → excellent. For natural photographic images with constantly varying intensities → RLE may INCREASE file size (if every pair is different). JPEG does NOT use raw RLE — it uses RLE after DCT+quantization, applied to the zigzag-scanned AC coefficients.

📝 RTU PYQs — RLE
Q. Write notes on Run-Length encoding and Shift codes.
May 2025 · Part B Q5 · 4 marks
Run-Length Encoding (RLE):
RLE compresses consecutive sequences of identical symbols. A sequence of n identical pixels with value v is encoded as the pair (v, n).

Example: Binary image row: 11111100001111
RLE: (1,6), (0,4), (1,4) — 6 numbers instead of 14

Applications: Fax machines (ITU-T T.4), BMP file format, PCX files, initial step in JPEG (for AC coefficients)
Best case: Long runs → high compression
Worst case: Alternating values → expansion (1,1),(0,1),(1,1),...

Shift Codes:
A variable-length RLE variant where run lengths are encoded with shift codes. Short runs (most common) get short codes; long runs get longer codes. Essentially Huffman coding applied to the run-length values.
Example: run of 1→code "0"; run of 2→code "10"; run of 3→code "110"; run of 4→code "1110" etc.
This further reduces the number of bits compared to fixed-length RLE representation.

Lossy Compression Techniques

75% · Part B/C

What is Lossy Compression?

In lossy compression, the reconstructed image f̂(x,y) is not identical to the original f(x,y). Some information is permanently discarded — specifically, psychovisually redundant data that the HVS cannot perceive well.

Trade-off: Higher compression ratio in exchange for some quality loss.

1. Quantization (Core of Lossy Compression)

Mapping a large range of values to a smaller set. The most fundamental lossy operation.

Scalar Quantization
q(x) = round(x / Δ) × Δ
Δ = quantization step size. Larger Δ → more compression, more error.

Vector Quantization (VQ): Instead of quantizing individual pixel values, quantize blocks of pixels by mapping them to a nearest "codeword" in a codebook. Better compression than scalar quantization at same quality.

2. Predictive Coding (DPCM)

Exploit interpixel redundancy by predicting each pixel from its neighbors and encoding only the prediction error (residual). Errors are much smaller than original values → require fewer bits.

DPCM (Differential Pulse Code Modulation)
Error: e(x,y) = f(x,y) - f̂(x,y)
f̂ = predicted value (e.g., previous pixel). Only e(x,y) is encoded and transmitted.

If pixels are correlated, prediction errors are small → small values → more compressible. Used in JPEG-LS (lossless variant) and as first step in video compression.

3. Transform Coding (DCT/Wavelet)

Transform image to frequency domain, then apply coarser quantization to high-frequency coefficients (which contribute little to perceived quality).

Steps: Partition → Transform (DCT/DWT) → Quantize → Entropy code

This is the foundation of JPEG (DCT) and JPEG 2000 (Wavelet).

4. Fractal Compression

Based on the observation that different parts of a natural image are self-similar at different scales. Encode image as a set of fractal transformations (Iterated Function Systems) that reproduce the image when applied repeatedly.

Advantage: Very high compression, resolution-independent decoding.

Disadvantage: Very slow encoding (searching for self-similar parts is computationally expensive).

🔗
Connection: Lossy Compression ↔ Unit 3 (DCT) & Unit 4 (Quantization)

Lossy compression directly combines Unit 3's DCT (frequency domain transform) with quantization (Unit 4's degradation concept in reverse — here we deliberately introduce degradation for compression). The quantization table in JPEG is specifically designed using psychovisual sensitivity data — larger step sizes for high frequencies (less sensitive) and smaller step sizes for low frequencies (more sensitive).

JPEG Compression

🔥 90% · Part C

Overview

JPEG (Joint Photographic Experts Group) is the most widely used lossy image compression standard. It uses the Discrete Cosine Transform (DCT) as its core compression tool, exploiting both interpixel and psychovisual redundancy.

Why DCT instead of DFT?

  • DCT coefficients are real (not complex) — simpler arithmetic
  • DCT has better "energy compaction" — most image energy concentrates in fewer low-frequency DCT coefficients
  • DCT eliminates boundary discontinuity artifacts better than DFT
  • Faster algorithms available (related to FFT)

JPEG Compression Pipeline

Fig 9.1 — JPEG compression pipeline: 6 stages from image to compressed bitstream

Stage-by-Stage Explanation

Stage 1: Color Space Conversion (for color images)

RGB → YCbCr. The Y (luminance) channel is compressed less than Cb/Cr (chrominance) channels because the HVS is more sensitive to luminance detail. Chrominance channels are often subsampled 2:1 (4:2:0 or 4:2:2 subsampling).

Stage 2: Block Splitting + Level Shifting

Image is divided into non-overlapping 8×8 pixel blocks. Each pixel is level-shifted by subtracting 128 (for 8-bit images), centering values around 0 (range: -128 to +127). This improves DCT performance.

Stage 3: 2D Discrete Cosine Transform (DCT)

2D DCT Formula for 8×8 block
F(u,v) = (1/4)·C(u)·C(v) · Σₓ Σᵧ f(x,y)·cos[(2x+1)uπ/16]·cos[(2y+1)vπ/16]
C(u) = 1/√2 if u=0, else 1. x,y = spatial coordinates (0–7). u,v = frequency coordinates (0–7).
F(0,0) = DC coefficient (average of block). F(u,v) for u,v>0 = AC coefficients (variations).

After DCT, most image energy concentrates in low-frequency coefficients (top-left of F(u,v) matrix). High-frequency coefficients are near zero for smooth image regions.

Stage 4: Quantization (Lossy Step!)

Quantization
F̂(u,v) = round[ F(u,v) / Q(u,v) ]
Q(u,v) = quantization step from JPEG standard quantization table. Larger Q → coarser quantization → more loss.

The JPEG quantization table has larger values for high frequencies (more aggressive quantization of high-frequency detail, which is less visible to the HVS) and smaller values for low frequencies.

📌 JPEG Standard Luminance Quantization Table (partial)
  16  11  10  16  24  40  51  61
  12  12  14  19  26  58  60  55
  14  13  16  24  40  57  69  56
  14  17  22  29  51  87  80  62
  18  22  37  56  68 109 103  77
  ...

Notice: top-left (low freq) = 16; bottom-right (high freq) = up to 109. High-frequency DCT coefficients get divided by much larger values → rounded to zero → compressed away.

Stage 5: Zigzag Scanning

The 8×8 quantized DCT matrix is scanned in a zigzag pattern — starting from DC coefficient (0,0), going through frequencies in order of increasing frequency. This groups the zeros (which come from high-frequency quantization) together at the end of the 64-element sequence.

Stage 6: Entropy Coding (Lossless)

The DC coefficient of each block is encoded using DPCM (difference from previous block's DC). The AC coefficients are encoded using RLE (to compress runs of zeros) followed by Huffman coding.

Decompression (JPEG Decode)

Reverse all steps: Entropy decode → inverse zigzag → dequantize (multiply by Q) → IDCT → add 128 → reconstruct image. The quantization step causes irreversible loss — dequantized values are only approximations.

Dequantization
F_approx(u,v) = F̂(u,v) × Q(u,v)
This is an approximation of the original F(u,v) — the quantization error (F − F_approx) is the lossy part.
⚡
Why DCT? — The Two Reasons RTU Asks

1. Energy compaction: DCT packs most image energy into a few low-frequency coefficients. After quantization, many high-frequency coefficients become zero → highly compressible by RLE.
2. No complex numbers: Unlike DFT, DCT produces only real-valued coefficients — faster computation, simpler hardware. This is why the JPEG standard chose DCT over DFT.

Transform-Based Image Compression (JPEG):

Block Diagram:
[Draw the 6-stage pipeline from Fig 9.1: Image → Block Split → DCT → Quantize → Zigzag → Entropy Code → JPEG file]

Stage 1 — Partitioning: Image divided into 8×8 blocks. Pixel values level-shifted by −128 to center around zero.

Stage 2 — DCT (2D): Each 8×8 block transformed to frequency domain using 2D-DCT. Produces 64 coefficients: 1 DC (average) + 63 AC (variation). Most energy concentrated in few low-frequency coefficients.

Stage 3 — Quantization (Lossy): F̂(u,v) = round[F(u,v) / Q(u,v)]. JPEG quantization table Q uses larger divisors for high frequencies. Result: many high-frequency AC coefficients → 0. This is the ONLY lossy stage.

Stage 4 — Zigzag Scan: 8×8 matrix linearized in zigzag order (DC→low freq→high freq). Groups trailing zeros together for efficient RLE.

Stage 5 — Entropy Coding: DC: DPCM between adjacent blocks. AC: RLE encodes (zero_count, nonzero_value) pairs + Huffman coding. EOB (End of Block) marker for remaining zeros.

Two reasons for choosing DCT in JPEG:
1. Superior energy compaction: DCT concentrates 90%+ of block energy in ~10% of coefficients (the DC and low-frequency ACs). This allows aggressive zeroing of high-frequency coefficients with minimal perceptual quality loss, enabling high compression ratios (10:1 to 100:1).

2. Real-valued output (computational efficiency): Unlike DFT which produces complex coefficients, DCT produces only real-valued coefficients. This makes hardware implementation faster and simpler. DCT can be computed using fast FFT-like algorithms with only real arithmetic — critical for real-time JPEG encoding in cameras and smartphones.
